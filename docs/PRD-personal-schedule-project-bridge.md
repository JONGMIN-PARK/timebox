## 1. 개요 (Overview)

### 1.1 배경

TimeBox는 **개인** 일정(캘린더, 할 일, 타임박스, 리마인더, D-Day)과 **프로젝트** 협업(멤버, 칸반, 간트, 파일, 채팅, 인박스)이 공존한다. 다만 데이터 모델상 개인 이벤트·개인 할 일은 `userId` 단위로만 존재하고, 프로젝트 태스크와 **명시적으로 연결되지 않아** 다음이 어렵다.

- “이 회의/마감이 어느 프로젝트 일인지”를 개인 캘린더에서 식별
- 프로젝트 관점에서 “멤버들의 마일스톤·일정”을 한 화면에서 파악
- 공유 범위(누가 무엇을 보는지)를 사용자가 직관적으로 이해

### 1.2 목표 (Goals)

| ID | 목표 |
|----|------|
| G1 | 개인 일정·할 일을 **선택적으로** 프로젝트(또는 워크 맥락)에 연결할 수 있다. |
| G2 | 프로젝트 참여자가 **프로젝트 단위 일정/마감**을 조회할 수 있는 뷰가 있다. |
| G3 | 프로젝트·팀 **가시성(visibility)** 정책이 제품 표면(UI·문구)과 API 동작에서 일치한다. |
| G4 | 초대·권한 흐름이 실사용에서 막히지 않도록 **초대 UX**를 보완한다. |
| G5 | (선택) 개인 일정과 **외부 캘린더** 간 연동으로 이탈을 줄인다. |

### 1.3 비목표 (Non-Goals) — 초기 범위에서 제외

- 실시간 동시 편집(Notion 수준 CRDT)
- 전사 SSO / 엔터프라이즈 전용 규정 준수 인증만을 위한 별도 스코프
- 프로젝트 없이 “가족/친구 그룹”만을 위한 별도 제품 라인

## 2. 사용자 · 시나리오 (Personas & Scenarios)

### 2.1 페르소나

- **P-개인**: 프리랜서·지식 worker. 개인 캘린더·할 일을 매일 쓰고, 가끔 팀 프로젝트에 참여.
- **P-PM**: 프로젝트 오너/관리자. 마감·회의·리소스를 프로젝트 단위로 보고 싶음.
- **P-멤버**: 프로젝트에 viewer/member. 본인 할 일과 프로젝트 이슈를 헷갈리지 않게 정리하고 싶음.

### 2.2 핵심 시나리오

1. **S1 연결**: 사용자가 개인 캘린더에 “클라이언트 A 미팅”을 만들 때, 프로젝트 “클라이언트 A”에 연결해 둔다. 나중에 프로젝트 대시보드에서 해당 기간 일정을 필터로 본다.
2. **S2 할 일 연동**: 개인 할 일 목록에서 항목에 프로젝트를 태그하면, 프로젝트 화면에서 “연결된 개인 할 일(본인만 전체 내용)” 또는 “마감만 요약”으로 볼 수 있다(정책은 4절에서 정의).
3. **S3 팀 캘린더**: PM이 프로젝트 탭에서 “프로젝트 캘린더”를 열어, **연결된 이벤트 + 프로젝트 태스크 due**를 한 타임라인에서 본다.
4. **S4 가시성**: 신규 멤버가 “내가 볼 수 있는 범위”를 설정 화면 또는 프로젝트 소개에서 이해한다.
5. **S5 초대**: 사용자명을 몰라도 **초대 링크** 또는 관리자가 **대기 초대**를 발행·취소할 수 있다.

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 개인 이벤트·할 일 ↔ 프로젝트 연결 (G1)

| ID | 요구사항 | 우선순위 | 비고 |
|----|----------|----------|------|
| F1.1 | 캘린더 이벤트에 선택적 `projectId`(nullable)를 둔다. | P0 | 마이그레이션, API, 클라이언트 폼 |
| F1.2 | 개인 할 일(todos)에 선택적 `projectId`(nullable)를 둔다. | P0 | 기존 trash·정렬·필터와 충돌 검토 |
| F1.3 | 연결된 프로젝트에 대해 **현재 사용자가 멤버(또는 팀 그룹 권한)**인 경우에만 `projectId` 설정·조회 가능. | P0 | 서버 검증 필수 |
| F1.4 | UI: 이벤트/할 일 생성·편집 시 프로젝트 선택 드롭다운(“없음” 기본). | P0 | |
| F1.5 | UI: 캘린더·할 일 목록에서 프로젝트 색/이름 배지 표시. | P1 | |
| F1.6 | 개인 데이터는 기본 **비공개**; 다른 멤버에게는 “바쁨/제목 마스킹” 등 정책은 4절에서 결정. | P1 | 최소 구현은 “타인에게 개인 이벤트 상세 비노출” |

**수용 기준 (예시)**

- 이벤트/할 일 저장 시 존재하지 않거나 접근 불가한 `projectId`면 400/403.
- `projectId` null인 기존 데이터는 그대로 동작(하위 호환).

### 3.2 프로젝트 단위 일정 뷰 (G2)

| ID | 요구사항 | 우선순위 | 비고 |
|----|----------|----------|------|
| F2.1 | 프로젝트 내 **캘린더(또는 일정)** 탭 또는 서브뷰를 제공한다. | P1 | 기존 탭 구조와 합칠지 별도 탭인지 UX 결정 |
| F2.2 | 뷰에 포함: (a) 해당 프로젝트에 연결된 **본인 이벤트**, (b) **프로젝트 태스크의 due date** (이미 스키마에 있다면). | P1 | (b)는 기존 API 재사용 우선 |
| F2.3 | (선택) 팀 정책이 허용 시, 다른 멤버의 “연결된 이벤트”를 **요약(제목 또는 바쁨만)**으로 표시. | P2 | 프라이버시 스위치와 세트 |
| F2.4 | 월/주 단위 전환은 개인 캘린더와 UX 일관. | P2 | |

**수용 기준 (예시)**

- 프로젝트 멤버가 아닌 사용자는 해당 프로젝트 일정 API 403.
- 빈 프로젝트도 뷰는 로딩·빈 상태 정상.

## 4. 정책 · 보안 (Policy & Security)

### 4.1 개인 이벤트/할 일의 타인 노출

- **기본안**: 다른 사용자에게는 **내용 미노출**. 프로젝트 “팀 캘린더”에는 내 **바쁨 슬롯만** 또는 **연결된 항목만 제목 공개** 중 제품이 하나를 선택해 일관 적용.
- **설정안(후속)**: 사용자별 “프로젝트에 연결된 일정을 팀에 제목까지 공개” 토글.

### 4.2 권한

- `projectId` 연결 시: 해당 프로젝트의 `projectMemberMiddleware`와 동일한 기준으로 읽기/쓰기 판단.
- iCal URL: **추측 불가 토큰**, 재발급 시 이전 URL 무효 옵션.

### 4.3 감사

- (선택) 프로젝트 연결 변경은 `user_activity_log`에 남김.

## 5. 기술 메모 (Technical Notes)

- **DB**: `events`, `todos`에 `project_id` nullable FK 또는 동등 무결성(서버 검증).
- **API**: 목록 조회 시 `?projectId=` 필터; 기존 클라이언트는 필터 없이 동작.
- **클라이언트**: 캘린더·할 일 폼에 공통 `ProjectPicker`(멤버십 있는 프로젝트만).
- **성능**: 월 범위 쿼리 인덱스 `(user_id, start_time)` 등 검토.

| Phase | 범위 | 산출물 |
|-------|------|--------|
| **Phase A** | F1.1–F1.4, F3.1–F3.2 | 스키마·API·최소 UI, visibility 문구 정합 |

## 7. 성공 지표 (Success Metrics)

- 프로젝트 활성 사용자 중 **이벤트/할 일에 project 연결 비율**(채택률).
- 프로젝트 **일정 뷰 주간 활성**(DAU/WAU 내 클릭 또는 체류).
- **초대 완료율**(링크 도입 시 기존 대비).
- 지원 티켓: “일정이 프로젝트와 안 맞는다” 유형 감소(정성).

## 8. 오픈 이슈 (Open Questions)

1. 개인 할 일을 프로젝트에 연결했을 때 **칸반 태스크로 복제/동기**할지, **링크만** 할지.
2. 팀 그룹 viewer가 볼 수 있는 데이터 범위를 프로젝트 오너가 좁힐 수 있는지.
3. 모바일에서 프로젝트 피커·일정 뷰 우선순위.
